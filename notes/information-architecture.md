# Information Architecture

## Purpose

Information architecture determines how users move through the archive, understand available materials, and find records. For public research archives, the structure needs to serve both casual visitors and researchers who need direct access to records, documents, and context.

## Core User Paths

A public archive may need to support users who want to:

- Understand the project and research context.
- Browse artifact records.
- Search for specific objects, documents, or terms.
- View returned artifacts or special categories.
- Read contextual pages.
- Access document libraries or source materials.
- Understand how the archive can grow over time.

## Example Site Sections

- Home.
- About / Project Context.
- Archive Browse.
- Search.
- Returned Artifacts.
- Document Library.
- Research or Context Pages.
- Contact or Credits.

## Design Principles

- Use clear navigation labels.
- Make search and browse paths easy to find.
- Separate archive records from contextual pages.
- Keep page templates consistent.
- Make record pages readable and scannable.
- Support both desktop and mobile browsing.
- Avoid hiding important project context behind too many clicks.

## Implementation Thinking

In Omeka S, information architecture connects several layers:

- Site pages and navigation.
- Item/resource templates.
- Browse/search behavior.
- Metadata labels.
- Theme layout and styling.
- Client maintenance needs.

Good information architecture helps non-technical maintainers keep the site coherent after launch.
